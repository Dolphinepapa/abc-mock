# 多角色账号系统 + CMO 监督路径

> 一个比较大的改动。引入三个角色 (CMO / Maya / Devon)、角色切换、CMO 监督面板、审批流和 Pattern 质疑流。建议分 Phase 推,每个 Phase 一新会话。

---

## 心智模型 (这部分 Claude Code 要先理解)

产品有三层角色,每层看到的东西不同:

```
CMO (L9 · 机密)
└─ 监督 + 审批战略级决策 + 质疑公司大脑方法论
   不做日常运营。默认 trust but verify。

Maya Chen (L7 · 敏感) VP eCommerce
└─ 直接管 SKU-A 落地灯 + 床架 SKU-117
└─ 3 个下属 report to her: Devon / Sara / Jamal
   能看自己的 conversation + 团队的 conversation (read-only)
   战略级决策需要 CMO 审批,日常决策自己决定

Devon Park (L6 · 内部) Sr Growth (Maya 下属)
└─ 直接管移动充电宝
   只能看自己的 conversation,看不到 Maya/Sara/Jamal 的
```

**关键产品原则**:

1. **权限隔离是真隔离**。Devon 切换进去,看不到任何其他角色的对话。不是 "filter" 是 "isolation"。
2. **CMO 默认信任团队**。所有 VP 级别决策默认通过,CMO 可走"监督路径"主动审查。需要 CMO 显式审批的只有"超出 VP 权限范围"的事 (比如大额预算追加)。
3. **质疑/审批触发的还是 chat + canvas**。坚持 "chat is the verb" 原则。CMO 点 [质疑] 不是弹个 modal,是开一个新 conversation thread。

## 不要做的事

- **不要给 Devon 视图加任何"你看不到的东西在这"的提示**。权限隔离应该是真隔离,Devon 视图里看不到 Maya 的活动,**也不应该看到"还有 X 个对话被隐藏"的提示**。Devon 不知道存在那些对话才是对的。
- **不要在 CMO 监督面板加分析图表(趋势线 / 健康度地图)默认展开**。监督面板是"待办 + 监督"形态,不是 dashboard。整体盘子那个 section 默认折叠是对的。
- **不要让 CMO 能直接修改 pattern 内容**。CMO 只能"质疑触发 audit",不能直接编辑。审计 + 治理的核心是**走流程**,不是越权操作。
- **不要让 [打回] 和 [质疑] 视觉上一样重**。打回 = 拒绝,语义重。质疑 = 讨论,语义轻。质疑应该是大部分 CMO 行为的入口。
- **不要给 CMO 视图加 chat input**。CMO 不发起 conversation,CMO 触发 [质疑] / [审批] 这种结构化动作。chat 输入框是给运营角色用的。
